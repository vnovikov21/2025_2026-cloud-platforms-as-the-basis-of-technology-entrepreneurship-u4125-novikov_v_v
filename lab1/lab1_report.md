## Цель работы
Ознакомиться с основными сервисами Google Cloud Platform, научиться работать с Service Accounts, создавать виртуальные машины (Compute Engine), работать с облачным хранилищем (Cloud Storage) и понимать важность правильного управления правами доступа (IAM).

## Ход работы

1. **Получение доступа**  
   Я заполнил Google-форму своей Gmail-почтой. После этого мне был предоставлен доступ к учебному проекту в Google Cloud.

2. **Установка Google Cloud CLI**  
   На свой компьютер с Windows 10 я установил утилиту `gcloud`. Это позволило в дальнейшем работать с Google Cloud из командной строки.

3. **Создание Service Account**  
   В консоли Google Cloud я перешёл в раздел **IAM & Admin → Service Accounts** и создал новый сервисный аккаунт:  
   - **Имя:** `vnovikov-sa-lab1`  
   - **Роль:** `Storage Admin` (полные права на работу с Cloud Storage).  

   Service Account — это специальная учётная запись, которая используется для автоматизации и работы приложений/сервисов.

4. **Создание виртуальной машины**  
   В разделе **Compute Engine → VM instances** я создал виртуальную машину со следующими параметрами:  
   - **Имя:** `vnovikov-vm-lab1`  
   - **Тип машины:** `e2-micro` (самый маленький и дешёвый вариант)  
   - **Режим:** `Spot` (дешёвый прерываемый экземпляр)  
   - Операционная система: Debian Linux (по умолчанию)  

   После создания подключился к машине через встроенный браузерный SSH.

5. **Работа с Cloud Storage**  
   На виртуальной машине с помощью команды `gsutil` я скопировал 3 файла из публичного бакета `lab1-bucket-itmo` в локальную папку:

   ```bash
   gsutil cp gs://lab1-bucket-itmo/* ./

Командой ls -lah проверил, что файлы действительно появились на виртуальной машине. Скриншот 1: Список файлов после копирования.

6.	**Изменение прав доступа (IAM)**
   Вернулся в консоль и изменил роль сервисного аккаунта vnovikov-sa-lab1:
   - Убрал роль Storage Admin
   - Добавил роль Compute Viewer (только просмотр информации о виртуальных машинах)
После этого снова попытался скопировать файлы из бакета на виртуальную машину. Результат: Появилась ошибка доступа (Permission denied).
Ошибка при попытке копирования после смены роли. Вывод: Права доступа очень важны. Один и тот же сервисный аккаунт может иметь совершенно разные возможности в зависимости от назначенных ролей.
7.	**Очистка ресурсов После выполнения работы я удалил:**
   - Виртуальную машину vnovikov-vm-lab1
   - Service Account vnovikov-sa-lab1
Это важно делать, чтобы не расходовать бесплатные кредиты проекта. Удалил все созданные ресурсы (VM и Service Account).

## Выводы
Google Cloud удобная платформа. Важно правильно управлять правами доступа (IAM) и не забывать удалять ресурсы после работы, чтобы не тратить деньги.
