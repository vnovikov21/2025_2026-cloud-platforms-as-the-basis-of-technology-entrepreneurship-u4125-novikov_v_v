## Цель работы
Разработать схему инфраструктуры MVP AI-приложения на Google Cloud Platform, спроектировать три стадии развития продукта (Initial, Testing, Production), рассчитать примерную экономическую модель и обосновать выбор используемых сервисов.

## Описание приложения
Приложение представляет собой **AI-ассистента** — интеллектуальный чат-бот, который помогает пользователям генерировать тексты, идеи для бизнеса, анализировать данные, а также работать с изображениями с помощью современных моделей искусственного интеллекта (Google Gemini и Vertex AI).

**Предполагаемая нагрузка:**
- **Initial (MVP)**: до 100 активных пользователей в сутки
- **Testing (с партнёрами)**: до 1000 активных пользователей в сутки
- **Production**: от 5000+ активных пользователей в сутки

Такое приложение может использоваться в образовании, маркетинге, разработке контента или как внутренний инструмент компании.

## Архитектура инфраструктуры

### 1. Стадия 1 — Initial (MVP)

На данном этапе важно максимально быстро и дёшево запустить продукт, чтобы проверить гипотезу.

```text
                     ┌─────────────────────────────┐
                     │        Пользователь         │
                     └──────────────┬──────────────┘
                                    │ HTTPS
                                    ▼
                     ┌─────────────────────────────┐
                     │     Cloud Run (Frontend + Backend)    │
                     └──────────────┬──────────────┘
                                    │
            ┌───────────────────────┼────────────────────────┐
            ▼                       ▼                        ▼
┌────────────────────┐   ┌────────────────────┐   ┌────────────────────┐
│   Gemini API       │   │   Firestore DB     │   │  Cloud Storage     │
│  (AI модель)       │   │   (NoSQL)          │   │   (файлы, фото)    │
└────────────────────┘   └────────────────────┘   └────────────────────┘

На первом этапе весь функционал (и фронтенд, и бэкенд) размещается в одном сервисе Cloud Run. Для хранения данных используется простая и быстрая база Firestore, а файлы пользователей хранятся в Cloud Storage. Связь с AI происходит через Gemini API. Это самое дешёвое и быстрое решение.

Добавляется Load Balancer для распределения нагрузки. Появляется возможность запускать две версии бэкенда одновременно (A/B-тестирование). База данных заменяется на более мощную Cloud SQL (PostgreSQL). Это позволяет собирать обратную связь от партнёров без риска потери данных.

## Экономическая модель

| Стадия              | Основные сервисы                              | Примерная стоимость в месяц | Обоснование |
|---------------------|-----------------------------------------------|-----------------------------|-----------|

*Цены ориентировочные и зависят от региона и фактического использования.*

## Обоснование выбора сервисов

- **Cloud Run** выбран как основной вычислительный сервис, потому что он serverless — не нужно управлять серверами, а оплата идёт только за реальное время работы.
- **Firestore → Cloud SQL** — постепенный переход от простой NoSQL-базы к полноценной реляционной базе данных.
- **Vertex AI / Gemini** — самые современные AI-модели от Google с удобной интеграцией.
- **Load Balancer + CDN** — обеспечивают высокую доступность и скорость работы для пользователей по всему миру.
- **Memorystore (Redis)** — необходим для кэширования и высокой производительности на этапе Production.
